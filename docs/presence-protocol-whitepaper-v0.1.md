# Presence Protocol Whitepaper v0.1

## MyShape Protocol — The Sovereign Presence Layer for the AI Era

**Status**: Working Draft  
**Authors**: MyShape Protocol  
**Date**: 2026-06-23

## Abstract

The Presence Protocol defines a cryptographic framework for verifying human presence — not identity, not credentials, not biometrics — in an era where AI can forge every static signal. The protocol introduces a novel security model based on **economic asymmetry**: rather than claiming motion is "unforgeable," it demonstrates that forgery costs can be engineered to exceed attack rewards for any realistic adversary. The system combines five defense layers — Motion Signature, Device Attestation, Context Graph, Temporal Continuity, and CoPresence — with dynamic risk-based weighting that adapts to both transaction value and cumulative risk exposure. This whitepaper presents the complete threat model, challenge-response protocol, motion signature mathematics, multi-factor verification framework, and experimentation roadmap that together define the Presence Protocol v1.0.

## 1. Introduction: Why Presence, Why Now

### 1.1 The Collapse of Identity Systems

All existing identity systems — email, password, face, fingerprint, wallet, KYC — prove one thing:

> "This account belongs to you."

They prove nothing about:

> "The human operating this device right now is you."

As AI systems achieve indistinguishability from humans in static modalities (text, image, voice, face), every credential-based identity system faces existential collapse. A bot can create 10,000 accounts in minutes. A human cannot create 50. The asymmetry is fatal.

### 1.2 Presence as the Last Sovereign Signal

Among all observable human signals, only one class possesses the necessary properties:

| Property | Static Biometrics | Motion (Presence) |
|:---|:---|:---|
| Real-time generated | ❌ | ✅ |
| High entropy | ❌ (low-dimensional) | ✅ (high-dimensional) |
| Unpredictable | ❌ | ✅ |
| Non-replayable | ❌ | ✅ |
| Non-replicable | ❌ | ✅ |
| Non-storable | ❌ (templates exist) | ✅ (no template) |

**Motion is the last sovereign signal.** It is not a record. It is not a template. It is a continuous, high-dimensional, real-time field generated by the irreducible physics and biology of the human body.

### 1.3 The MyShape Thesis

> Identity becomes a field, not a file.

MyShape does not store motion. MyShape does not transmit motion. MyShape does not reconstruct motion. It extracts a cryptographic proof from motion geometry, verifiable anywhere, without revealing the body.

### 1.4 What This Protocol Proves

- **Not**: "You are a human." (World / Proof-of-Human)
- **But**: "You — this specific entity — are present right now."

This is the higher layer. World proves membership in the human set. MyShape proves singularity within that set.

## 2. Security Philosophy

### 2.1 Not "Cannot Be Forged" — "Not Worth Forging"

The Presence Protocol's security model is **economic**, not absolutist:

> Security is not physical invincibility. Security is engineered cost asymmetry.

| | Verification | Forgery |
|:---|:---|:---|
| Cost | ~$0.001 per check | $10K–$500K+ |
| Time | <100ms | Days to months |
| Scalability | Unlimited | Per-target, non-scalable |

As long as:

$$\text{Attack Cost} \gg \text{Expected Gain}$$

the system is secure — even if forgery is theoretically possible.

### 2.2 The Core Security Axiom

> Under the most pessimistic AI assumptions (2030+), pure passive motion recognition collapses. But active challenge-response with multi-factor verification can maintain the attack cost above the economic threshold for any target value tier.

This is not a claim of invincibility. It is a claim of **engineered infeasibility**.

## 3. Threat Model

### 3.1 Participants

| Role | Description | Trust |
|:---|:---|:---|
| **User** (Human Subject) | The entity whose presence is being verified | Body = root of trust |
| **Verifier** | Entity initiating challenge and consuming verification result | Honest-but-curious |
| **MyShape Node** | Protocol execution environment (local SDK + remote verification nodes) | Partially trusted |
| **Attacker** | Any entity attempting to forge presence | Untrusted |
| **AI Adversary** | Attacker with strong multimodal generative capability (2028+ level) | Untrusted |
| **Data Harvesting Adversary** | Entity systematically collecting motion data to reduce future forgery costs | **Silent systemic threat** |

### 3.2 Trust Boundaries

| Layer | Trust Level | Threat |
|:---|:---|:---|
| TEE / Secure Enclave | High | Side-channel, supply chain |
| OS | Medium-Low | Root/jailbreak → full control |
| App Sandbox | Medium | Escapable post-jailbreak |
| Camera Driver | Low | Hookable, injectable |
| CMOS Sensor Hardware | High | Physical replacement / optical spoofing |

**Critical**: The trust boundary for motion capture must be pushed below the application layer — ideally into TEE or, at minimum, to sensor hardware level.

### 3.4 Attacker Cost Model

| Time | B2a (AI Forgery) | B3 (Device Attack) | Attacker's Optimal Path |
|:---|:---|:---|:---|
| **T₀** (2026) | $10K–$200K | $1K–$50K (with physical access) | B3 if physical access; B2a otherwise |

**Core insight**: By 2030, software AI forgery may cost less than hardware device attacks. The protocol must defend both paths simultaneously. Defending only one means attackers flow through the other.

### 3.5 Pessimistic AI Assumptions

We explicitly assume the attacker possesses:

- Strong multimodal model (2028+ level)
- Abundant 2D video data of the target (public / stolen)
- Moderate 3D pose data (from AR/VR/motion capture leakage)
- Ability to fit a target's motion parametric model from sparse data
- Ability to generate real-time 3D motion sequences matching challenge requirements

We do **not** assume "AI cannot do this." We assume "AI can do this, but the cost and complexity are high." This makes the subsequent cost curves meaningful.

## 4. Presence Challenge Protocol

### 4.1 Design Objectives

The challenge protocol must satisfy:

| Requirement | Rationale |
|:---|:---|
| **Unpredictability** | AI cannot pre-compute or cache responses |
| **Real-time constraint** | AI cannot generate high-fidelity response within 100ms window |
| **Multi-modal** | Camera + IMU + temporal + kinematic |
| **High entropy** | Challenge space 10⁶–10⁹, impossible to enumerate |
| **Identity sensitivity** | Skeletal ratios, tremor spectrum, acceleration curves must naturally manifest |
| **Composable verification** | Future factors (device, context, co-presence) can be added |

### 4.2 Challenge Space Taxonomy

| Class | Example | Entropy | AI Difficulty |
|:---|:---|:---|:---|
| Single joint | "Raise hand" | Low | Low — easily generated |
| Multi-joint coordinated | "Right hand circle + left hand still" | Medium | Medium — multi-joint coordination needed |
| Non-periodic trajectory | "Draw an irregular triangle" | High | High — non-periodic generation is hard |
| Bimanual asynchronous | "Right hand CCW circle + left hand square" | High | High — dual frequency generation |
| **Full-body coordinated** | "Right hand circle + torso lean 10° left + head level" | **Maximum** | **Maximum** — full kinematic chain consistency |

**Protocol requirement**: Every challenge must contain **at least 2–3 joint groups + non-periodic trajectory + asynchronous motion**.

### 4.3 Coupling Constraint (Patch 1)

A challenge composed of independently-generatable sub-actions (e.g., "right hand circle + left hand square") is vulnerable to parallel generation with skeletal fusion.

**Requirement**: Every challenge MUST contain at least one **shared kinetic chain constraint** — sub-actions that share a body segment, such that independent generation followed by fusion produces detectable kinematic inconsistency.

Example of coupled challenge:
> "Right hand draws a circle while the torso leans 10° right and the head maintains forward gaze."

The right arm trajectory, torso lean, and head stabilization share the spinal kinetic chain. They cannot be independently generated and linearly blended without introducing unnatural torque patterns detectable at verification.

### 4.4 Challenge Entropy

$$\text{Challenge Entropy} = |\text{Action Space}| \times \text{Complexity} \times \text{Temporal Dimension}$$

| Entropy Source | Contribution |
|:---|:---|
| Action combination | 50³ = 125,000 (3-action combos from 50-base library) |
| Parameter space | Continuous: direction (360°), amplitude (0–50cm), speed (0.1–2.0 m/s), duration (0.2–2.0s) |
| Temporal entropy | Random start time, random pauses, random rhythm |
| Random perturbation | Final pause duration, trajectory deviation |

**Target**: Challenge entropy at 10⁶–10⁹, making pre-computation infeasible.

### 4.5 Timing Constraints

| Parameter | Value | Rationale |
|:---|:---|:---|
| Challenge delivery → response start | ≤ 100ms | Below AI real-time generation latency |
| Total action duration | 3–5 seconds | Sufficient for kinematic feature extraction |
| Challenge delivery | Single atomic message | No incremental challenge — prevents partial generation |
| Challenge issuance | Immediately before execution | No pre-computation window |

Current AI Text-to-Motion latency: 500ms–2s. Future (2030) projection: 100–200ms. The 100ms window is designed to stay ahead of the trend line.

### 4.6 Challenge Composition

A challenge MUST contain:

1. **Primary Action** — e.g., "Draw a counterclockwise circle with your right hand"
2. **Secondary Action** — e.g., "Touch your nose with your left index finger"
3. **Posture Constraint** — e.g., "Keep your body upright"
4. **Timing Constraint** — e.g., "Hold for 0.5 seconds at the apex"
5. **Velocity Constraint** — e.g., "Perform at moderate speed"
6. **Random Perturbation** — e.g., "Pause for 0.2 seconds before finishing"

Minimum: **4–6 sub-constraints per challenge**.

### 4.7 Unpredictability Guarantees

- Challenge generated by Verifier using CSPRNG
- Optional: generated inside TEE to prevent hook interception
- Delivered 100–200ms before execution start
- Single atomic message, no incremental disclosure
- Contains random perturbation unique to this session

### 4.8 Verification Interface

The verifier checks:

1. **Trajectory match** — shape, direction, amplitude, speed
2. **Kinematic consistency** — acceleration curves, jerk, joint angle continuity
3. **Tremor spectrum** — 8–12 Hz physiological tremor presence
4. **Multi-joint coordination** — cross-joint correlation matrix
5. **Cross-sensor consistency** — Camera trajectory vs. IMU acceleration
6. **Temporal consistency** — natural human latency, absence of robotic precision
7. **Historical consistency** — no anomalous convergence pattern (temporal drift detection)

## 5. Motion Signature Function

### 5.1 Design Objectives

| Requirement | Description |
|:---|:---|
| **Uniqueness** | Inter-person distance >> intra-person distance |
| **Temporal stability** | Natural variance bounded and modelable |
| **Forgery resistance** | AI cannot match deep kinematic features |
| **Cross-sensor consistency** | Camera + IMU must be physically consistent |
| **Composability** | Integrable with Device / Context / CoPresence factors |

### 5.2 Mathematical Structure

$$\text{MS}(t) = f\big(K(t), A(t), J(t), T(t), J_{\text{spec}}(f), C(t), S(t)\big)$$

Where:

| Symbol | Feature | Dimension | Forgery Difficulty |
|:---|:---|:---|:---|
| K(t) | 3D keypoint sequence (33–68 points) | N × 3 | Medium |
| A(t) | Acceleration (2nd derivative of K) | N × 3 | High |
| J(t) | Jerk (1st derivative of A) | N × 3 | **Very High** |
| T(f) | Position tremor spectrum (FFT of K) | Frequency bins | **Very High** |
| J_spec(f) | Jerk spectrum (FFT of J) | Frequency bins | **Extreme** |
| C(t) | Multi-joint coordination matrix | N × N | High |
| S(t) | Camera–IMU consistency | Scalar | **Extreme** |

Final signature: $\mathbf{z} = g(\text{MS}) \in \mathbb{R}^{512}$ via dimensionality reduction (PCA / Autoencoder / Transformer Encoder).

### 5.3 Kinematics K(t)

$$K(t) = \{(x_i(t), y_i(t), z_i(t))\}_{i=1}^{N}$$

N = 33 (MediaPipe Pose) or 68 (extended model).

**Skeletal Ratios**: Each person's bone length ratios are fixed and unique:

$$r_{ij} = \frac{\|p_i - p_j\|}{\|p_k - p_l\|}$$

AI has difficulty reproducing true skeletal proportions in 3D, especially depth-aware ratios.

### 5.4 Acceleration A(t)

$$A(t) = \frac{d^2 K(t)}{dt^2}$$

Human acceleration exhibits: smoothness, natural latency, muscle-driven characteristics. AI-generated acceleration tends toward: discontinuity, excessive perfection, absence of biological noise.

### 5.5 Jerk J(t)

$$J(t) = \frac{dA(t)}{dt} = \frac{d^3 K(t)}{dt^3}$$

Jerk is the most unforgeable kinematic dimension. Human jerk contains natural "jitter," muscle control differential signatures, and neural system latency patterns. AI jerk is typically discontinuous, excessively smooth, or lacks natural noise structure.

### 5.6 Jerk Spectrum J_spec(f) (Patch 2)

$$\mathbf{J}_{\text{spec}}(f) = \mathcal{F}\big(J(t)\big)$$

The frequency-domain representation of jerk reveals the **control system fingerprint** — the neural + muscular response in the frequency domain. This is deeper than position tremor:

| Spectrum | What it captures | AI Difficulty |
|:---|:---|:---|
| Position tremor T(f) | Mechanical oscillation at joints | Very high |
| **Jerk spectrum J_spec(f)** | **Neural control + muscle response dynamics** | **Extreme** |

Human jerk spectrum exhibits rich non-stationary structure in 1–15 Hz. AI-generated jerk spectrum typically shows only 0–3 Hz (over-smoothed) or full-band white noise (random jitter). J_spec(f) is elevated to a **first-class citizen** feature in the Motion Signature.

### 5.7 Tremor Spectrum T(f)

$$T(f) = \mathcal{F}\big(K(t)\big)$$

Target frequency bands:

| Band | Source | Individual Stability |
|:---|:---|:---|
| 3–5 Hz | Posture micro-adjustment | Medium |
| **8–12 Hz** | **Physiological tremor** | **High** |
| 20–30 Hz | Muscle micro-vibration | Medium-High |

These bands are individually stable, difficult to extract from video alone, and virtually absent from AI-generated motion.

### 5.8 Coordination Matrix C(t)

$$C(t) = \text{corr}\big(K_i(t), K_j(t)\big)$$

Captures inter-joint synergy, whole-body coordination, and kinetic chain structure. AI commonly exhibits: locally correct actions with globally incorrect coordination — phase relationships between joints that are physically unnatural.

### 5.9 Sensor Consistency S(t)

$$S(t) = \|A_{\text{camera}}(t) - A_{\text{IMU}}(t)\|$$

Camera-derived acceleration must match IMU-measured acceleration. Screen-based forgery cannot pass this check, as a display cannot simultaneously deceive both camera (optical) and IMU (inertial) with physically consistent signals.

### 5.10 Verification Scoring

$$\text{score} = \text{sim}(\mathbf{z}_{\text{user}}, \mathbf{z}_{\text{response}})$$

With threshold $\tau$ set per risk tier.

## 6. Multi-Factor Presence Verification

### 6.1 Design Objective

Single-factor security is insufficient. The protocol's defense-in-depth model:

> Motion is necessary but not sufficient. True security comes from factor composition.

Attack cost must rise **exponentially** with factor count, not linearly.

### 6.2 Overall Structure

$$\text{PresenceScore} = \mathcal{F}\big(w_M \cdot M,\ w_D \cdot D,\ w_C \cdot C,\ w_T \cdot T,\ w_P \cdot P\big)$$

| Factor | Symbol | Layer | What it defends against |
|:---|:---|:---|:---|
| Motion Signature | M | Biological | B2a (AI forgery) |
| Device Attestation | D | Hardware | B3 (sensor spoofing) |
| Context Graph | C | Behavioral | Anomalous usage patterns |
| Temporal Continuity | T | Temporal | Gradual approximation attacks |
| CoPresence | P | Network | Isolated forgery |

### 6.3 Factor 1: Motion Signature (M)

From Section 5. The biological layer — AI's hardest target.

### 6.4 Factor 2: Device Attestation (D)

$$D = f(\text{TEE}, \text{SE}, \text{OS Integrity}, \text{Sensor Fingerprint})$$

| Sub-factor | Description | Attack difficulty |
|:---|:---|:---|
| **TEE/SE integrity** | Verify device not jailbroken/rooted | Requires TEE exploit (+$500K) |
| **CMOS fingerprint** | Each sensor's noise pattern is unique and physical | Requires Level 2 hardware spoof |
| **Multi-sensor cross-validation** | Camera + IMU must agree | Single-sensor injection → must upgrade to multi-sensor sync injection |
| **Device risk tier** | Low-end devices → low-value ops only | Eliminates arbitrage |

**Device risk tiers:**

| Tier | Requirement | Max Operation Value |
|:---|:---|:---|
| Tier 1 | Any device with camera | $1K |
| Tier 2 | Device with TEE + IMU | $100K |
| Tier 3 | Tier 2 + verified secure boot chain | $1M+ |

### 6.5 Factor 3: Context Graph (C)

$$C = f(\text{Location}, \text{Device History}, \text{Behavior Pattern}, \text{Risk Profile})$$

| Sub-factor | What it detects |
|:---|:---|
| **Geographic continuity** | Beijing → New York in 1 hour = impossible |
| **Device history** | Sudden device switch → elevated risk |
| **Behavioral pattern** | 3 AM emergency large transfer → anomaly |
| **Risk profile** | Cumulative risk exposure over time window |

### 6.6 Factor 4: Temporal Continuity (T)

$$T = f(\Delta\text{MS}, \text{Fatigue Curve}, \text{Daily Variance})$$

| Sub-factor | Real user | AI forger |
|:---|:---|:---|
| **ΔMS stability** | Natural fluctuation | Anomalous convergence |
| **Fatigue curve** | Acceleration decay across repetitions | No natural fatigue |
| **Daily variance** | Natural day-to-day variation | Excessive consistency |

This is the primary defense against the **temporal drift attack** — an attacker gradually improving forgery fidelity across sessions.

### 6.7 Factor 5: CoPresence (P) — Future

$$P = f(\text{Multi-Device}, \text{Multi-Human}, \text{Spatial Proximity})$$

Requires simultaneous verification from multiple trusted entities. Not required for MVP; designated as the network-effect layer for v1.2+.

### 6.8 Dynamic Risk-Adaptive Weighting (Patch 3)

Weights are not static. They adapt to both **single-operation value** AND **cumulative recent risk exposure**:

$$w_i = w_i\big(\text{Value},\ \text{RecentRiskLoad}(t)\big)$$

$$\text{RecentRiskLoad}(t) = \sum_{k} \text{Value}_k \cdot e^{-\lambda (t - t_k)}$$

Where $\lambda$ is the decay constant (default: 15-minute half-life) and the sum is over all operations in the recent window.

This prevents **operation-splitting attacks**: 5 × $100K operations in 10 minutes ≈ 1 × $500K — weights automatically escalate.

| Operation Value | Recent Risk Load | M | D | C | T | P |
|:---|:---|:---|:---|:---|:---|:---|
| $10 | — | 0.7 | 0.2 | 0.1 | 0.0 | 0.0 |
| $1K | — | 0.5 | 0.3 | 0.2 | 0.0 | 0.0 |
| $100K | Low | 0.4 | 0.3 | 0.2 | 0.1 | 0.0 |
| $100K | **High** (5 recent) | 0.3 | 0.3 | 0.2 | 0.2 | 0.0 |
| $1M | — | 0.3 | 0.3 | 0.2 | 0.1 | 0.1 |
| $10M+ | — | 0.2 | 0.3 | 0.2 | 0.2 | 0.1 |

## 7. Experimentation Framework

### 7.1 Critical Questions

The protocol's scientific validity rests on five experiments:

| Experiment | Question | Criticality |
|:---|:---|:---|
| **E1** | Are Motion Signatures individually unique? | 🔴 Protocol collapses if false |
| **E2** | Are signatures temporally stable? | Required for T factor |
| **E3** | How close can AI get? At what cost? | 🔴 Core security assumption |
| **E4** | Do multi-factor costs rise exponentially? | Required for D factor design |
| **E5** | Do real attack costs match our curves? | Required for economic security proof |

### 7.2 E1: Uniqueness Test

- **Participants**: 100–500
- **Data per participant**: 50–200 motion sequences
- **Motion classes**: Single-joint, multi-joint, non-periodic, asynchronous, full-body
- **Metric**: Inter-person distance vs. intra-person distance
- **Target**: ROC-AUC > 0.98, EER < 1%

### 7.3 E2: Temporal Stability Test

- **Design**: Same participant, multiple sessions (morning/noon/evening, cross-day, fatigued/rested, different devices)
- **Metric**: $\Delta\text{MS}(t) = \|\text{MS}(t) - \text{MS}(t-\Delta t)\|$
- **Target**: Natural variance bounded, daily variance < inter-person variance

### 7.4 E3: AI Forgery Limit Test (CRITICAL)

- **Data tiers**: L1 (2D video), L2 (2D+depth), L3 (3D pose), L4 (motion capture)
- **Attack models**: Text-to-Motion, Motion Diffusion, 3D Pose Generator, Style Transfer, Real-time Generator
- **Metric**: $d_{\text{forgery}} = \|\text{MS}_{\text{fake}} - \text{MS}_{\text{real}}\|$
- **Expected**:
  - T₀: AI approaches 60–80%
  - T₁: AI approaches 80–90%
  - T₂: AI approaches 90–95%
  - **But**: fails on jerk spectrum, tremor, and IMU consistency at all tiers

### 7.5 E4: Multi-Factor Coupling Test

- **Test**: Break 1, 2, 3, 4 factors sequentially
- **Expected**: Single factor breakable; dual factor hard; triple factor extremely hard; quadruple factor uneconomical

### 7.6 E5: Attack Cost Simulation

- **Measure**: Data requirements, training cost, inference latency, generation quality, success rate
- **Validate**: $\text{AttackCost} \gg \text{ExpectedGain}$ across all value tiers

## 8. Security Guarantees

### 8.1 Formal Claims

1. **Unforgeability (economic)** — ∀ attacker A with budget B < ExpectedGain(target), Pr(A succeeds) < ε
2. **Non-replayability** — Replay success probability → 0 under nonce + timestamp + session binding
3. **Zero-knowledge** — Raw motion, feature vectors, and identity information never leave the device
4. **Composability** — Multi-factor score is strictly more discriminative than any single factor
5. **Privacy-preserving** — Verifier learns only "presence confirmed" + entropy score + timestamp

### 8.2 Future-Proofing (2030 Worst Case)

Under the most pessimistic AI assumptions:

| Assumption | Protocol Response |
|:---|:---|
| Universal motion foundation models exist | Challenge entropy + coupling constraints keep C_rt high |
| Individual fine-tuning costs collapse to ~$10 | Multi-factor makes pure motion insufficient alone |
| Real-time generation hits <100ms | Reduce challenge window, increase complexity |
| 3D sensor data widely leaked | Tremor + jerk spectrum + IMU consistency remain hard |

**The protocol's long-term security does not depend on any single assumption about AI capability. It depends on the engineered cost asymmetry across multiple coupled factors.**

### 8.3 Tiered Security Summary

| Operation Value | Required Factors | Attacker Budget Needed | Attack Economic? |
|:---|:---|:---|:---|
| <$1K | M + basic D | $10K+ | ❌ No |
| $1K–$100K | M + D + C | $50K+ | ❌ Usually no |
| $100K–$1M | M + D + C + T | $200K+ | ⚠️ Marginal |
| $1M+ | All 5 factors | $500K+ | ❌ Generally no |
| $10M+ | All 5 + maximum weights | $1M+ | ❌ No |

## A. Glossary

| Term | Definition |
|:---|:---|
| **Presence** | A continuous, real-time, high-entropy, non-replicable motion signal generated by a living human body |
| **Motion Vector (MV)** | High-dimensional representation of human motion over a fixed time window |
| **Motion Signature (MS)** | 512-dimensional vector encoding all kinematic, dynamic, and spectral features of a motion sequence |
| **Presence Entropy Score (PES)** | 0–1 score measuring biological entropy — 0 = artificial, 1 = maximally biological |
| **Proof of Presence (PoP)** | Cryptographic proof derived from Motion Signature, verifiable without exposing raw data |
| **ZK-Presence (ZKP)** | Zero-knowledge proof that a specific entity was present at a specific time |
| **Challenge** | Unpredictable, multi-constraint motion instruction issued by the Verifier |
| **Presence Score** | Weighted multi-factor score combining M, D, C, T, P |
| **RecentRiskLoad** | Time-decayed cumulative risk exposure used to adapt factor weights |
